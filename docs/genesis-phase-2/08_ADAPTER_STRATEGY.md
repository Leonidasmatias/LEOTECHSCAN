# 08 ‚Äî Adapter Strategy (Genesis Phase 2)

## Reference architecture: `services/geospatial/**`

Every rule below is extracted from, and validated against, the one subsystem in the repository that already practices this split successfully: `services/geospatial/coordinate-quality-engine.ts`/`brazil-bounds.ts`/`national-grid.ts`/`spatial-query-utils.ts`/`request-params.ts`/`compact-site.ts` (pure) plus `spatial-intelligence-engine.ts` (thin SQLite-touching adapter). The pattern exists specifically because Vitest could not collect a test file that transitively imports `node:sqlite` ‚Äî a concrete, repository-proven reason to keep pure logic pure, not an abstract preference.

## Adapter categories

1. **Entity adapters** ‚Äî translate legacy row data into canonical `EntityReference`/entity shapes (`02_CANONICAL_DOMAIN_MODEL.md`'s Site Entity Adapter is the first instance).
2. **Score adapters** ‚Äî translate a legacy engine's numeric output into canonical `Score` (Data Trust is first).
3. **Evidence adapters** ‚Äî translate legacy evidence arrays into canonical `Evidence[]`/`DataProvenance`.
4. **Recommendation adapters** ‚Äî translate/reconcile the three legacy recommendation shapes into canonical `Recommendation`.
5. **Persistence adapters** ‚Äî translate the Orchestrator's persistence decisions (`05_ORCHESTRATION_MODEL.md`) into actual SQLite writes, isolating every `INSERT`/`ensure*Tables` call behind one seam per table family.
6. **API projection adapters** ‚Äî translate canonical contract objects into HTTP JSON response shapes for new canonical routes, and (optionally, future) translate canonical objects back into legacy response shapes for a transition period (`13_MIGRATION_STRATEGY.md`).
7. **Legacy-engine adapters** ‚Äî the umbrella term for categories 2‚Äì4 when specifically wrapping an existing `services/*-engine.ts` file, as opposed to a category 1/5/6 adapter that has no single legacy engine counterpart.
8. **Graph-projection adapters** ‚Äî translate canonical entities into `sentinel-core`'s graph node/edge shapes, needed only once `sentinel-core/entities/*` stubs are implemented for real (`15_ARCHITECTURE_DECISIONS.md` ADR-013) ‚Äî not required for the first five adapters.

## Rules for every adapter (binding)

- **Adapters must not contain new business formulas.** A Score Adapter may compute `canonical = legacy / 100` (a documented, mechanical, universal conversion ‚Äî `02_CANONICAL_DOMAIN_MODEL.md`'s scale decision) but must never introduce a new weighting, threshold, or business rule not already present in the legacy engine it wraps. If the wrapped legacy output is wrong, the fix belongs in the legacy engine (or a Principle-2-sanctioned refactor of it), not smuggled into the adapter.
- **Adapters may translate shape and scale.** This is their entire job.
- **Adapters may attach metadata.** E.g. stamping `engineVersion`/`contractVersion`/`configurationVersion`, `executionMetadata`, `correlationId` ‚Äî all Orchestrator-supplied context, not computed by the adapter itself.
- **Adapters must preserve source values.** The original legacy 0‚Äì100 value, the original Portuguese recommendation string, etc. must remain retrievable (in `metadata`, in a `ScoreDriver.explanation`, or equivalent) ‚Äî never silently discarded in the translation, per `02_CANONICAL_DOMAIN_MODEL.md`'s scale-conversion rule.
- **Adapters must not silently repair invalid data.** If the legacy engine returns `null`/missing/out-of-range data, the adapter surfaces that as `INSUFFICIENT_DATA`/a `Limitation` (per the "unknown vs. zero" rule), never substitutes a guessed value. This mirrors `services/geospatial/coordinate-quality-engine.ts`'s own explicit "no auto-correction, suggestions only" rule, generalized to every adapter.
- **Adapters must expose limitations.** Every adapter output that wraps simulated/partial/sample-only legacy data (Copernicus, `sentinel-core`) must carry a `Limitation` reflecting that ‚Äî the Evidence Adapter in particular must never let `copernicus-truth.ts`'s `dataStatus: "simulated"` get lost in translation (`02_CANONICAL_DOMAIN_MODEL.md`'s Data Source invariant).
- **Adapters must be testable without SQLite when possible.** Categories 1‚Äì4 and 6 (translation-only) should be pure, taking already-fetched legacy data as input, following `spatial-intelligence-engine.ts`'s split ‚Äî fetch in a thin outer layer, translate in a pure inner function. Category 5 (persistence) and the DB-fetching half of categories 2‚Äì4 are the only parts allowed to touch `node:sqlite` directly, and even those should stay "thin" per the next rule.
- **Infrastructure-touching adapters must remain thin.** The DB-touching half of an adapter should do exactly two things: fetch legacy data, and call the pure translation function. No business logic, no scale conversion, no evidence construction inside the DB-touching half ‚Äî all of that lives in the pure half, testable in isolation, following `spatial-intelligence-engine.ts`'s own internal split between orchestration and `spatial-query-utils.ts`'s pure math.
- **Canonical models must not import legacy modules.** `services/intelligence/**` never imports from `services/*-engine.ts`, `sentinel-core/**`, or `node:sqlite`, directly or transitively ‚Äî this direction is enforced by the dependency graph (`12_DEPENDENCY_GRAPH.md`'s forbidden-dependencies list) and restates Principle 13.

## The first five adapters required for implementation (per mission Step 11 and cross-referenced against `14_IMPLEMENTATION_ROADMAP.md`)

1. **Site entity adapter** (`services/intelligence-adapters/site-entity-adapter.ts`) ‚Äî pure, `siteRow() ‚Üí EntityReference<"Site">`/`Site`. No DB access of its own (the caller already has the row). First because every other adapter needs an `EntityReference` to attach its output to.
2. **Data Trust Score adapter** (`services/intelligence-adapters/data-trust-score-adapter.ts`) ‚Äî thin DB-touching outer layer calling `dataTrustForSite(db, id, persist=false)` (note: **the adapter must call the legacy function with `persist=false`**, per Principle 6 ‚Äî the adapter's own preview mode must not inherit the legacy function's default side effect) plus a pure inner translator producing a `Score` with `type: "data-trust"`, `drivers` built from the eight confidence sub-scores plus the duplicate/alert penalties, `value = trustScore / 100`. Second because it is the most direct existing signal (Data Trust roadmap's own stated reasoning, independently corroborated by the pre-implementation audit).
3. **Evidence adapter** (`services/intelligence-adapters/evidence-adapter.ts`) ‚Äî wraps `evidence-center-engine.ts`'s five-item array into canonical `Evidence[]`, with `DataProvenance` populated from `import_audit` where traceable and an explicit `Limitation` where not (e.g. the Copernicus evidence item, which per rule must always declare `reliability` low and `origin.pipeline: "copernicus-truth:simulated"` or equivalent, never omit the simulated-data disclosure). Third because Recommendation generation needs Evidence to cite.
4. **Recommendation adapter** (`services/intelligence-adapters/recommendation-adapter.ts`) ‚Äî the reconciliation point for the three legacy recommendation shapes (Data Trust's string, `sentinel-core`'s objects, Evidence Center's embedded text); Phase 2.0 does not pre-decide which source wins in which situation, only that this adapter is where that decision is made, tested, and documented as its own ADR when built.
5. **API projection adapter** (`services/intelligence-adapters/api-projection-adapter.ts`) ‚Äî translates the above into the JSON shape served by the first new canonical route (`GET /api/intelligence/data-trust/site`, per `13_MIGRATION_STRATEGY.md`/`14_IMPLEMENTATION_ROADMAP.md` Increment 7), and is the only adapter category that is allowed to know about HTTP/Next.js request/response shapes ‚Äî none of adapters 1‚Äì4 should import anything from `next/server`.
