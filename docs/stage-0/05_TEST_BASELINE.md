# Stage 0 — Initial Automated Test Baseline (WP0.10)

## Framework

No test framework existed in this project before Stage 0. **Vitest** was chosen: it's the natural fit for a TypeScript/Next.js codebase with no existing test infrastructure to conflict with, needs no additional config beyond a path-alias mapping that mirrors `tsconfig.json`'s existing `"@/*"` alias, and runs plain `.test.ts` files with no framework-specific ceremony. Added to `package.json` as a devDependency (`vitest`) with a `"test": "vitest run"` script, and configured via `vitest.config.ts`.

## The tests

Originally 10 tests; test 9 (the Copernicus/Sentinel-1 truth check) was later split into two files as part of fixing the `node:sqlite` collection failure described below, bringing the total to 11 tests across 6 files. All of them use only in-memory data or static source-file text — no real Excel file, no real database, no network call, no filesystem write outside a single scratch path used and discarded within one test.

## Why these 10 and not some other 10

The exact original test list specified for WP0.10 was not fully preserved across a context boundary during this session (a compaction of the working conversation). Rather than guess at wording that might not match, this set was constructed directly from what Stage 0 actually built: one or more tests per work package that has runtime-checkable logic (WP0.5, WP0.6, WP0.7, WP0.4, WP0.3/WP0.12). WP0.8 and WP0.9 are deliberately **not** re-tested here with unit tests — they already have thorough, evidence-based integration test coverage documented in `04_IMPORT_SAFETY.md` and validated directly in `07_ROLLBACK_PLAN.md`'s backup/restore section, run against real (copied) SQLite databases in ways a mocked unit test would not meaningfully improve on.

## Verification status: real, but not through the tool this doc names

`npm install` could not be run in the cloud sandbox during this session — the npm registry returned `403 Forbidden` for every package (tried directly with `npm`, and with `bun` as an alternative; both blocked identically, ruling out a client-specific issue). This means `vitest` itself was never actually executed in this environment.

To avoid shipping test files whose assertions were never actually checked against the real code, every test's assertions were independently verified by executing the actual source files directly, using Node 22's native `--experimental-strip-types` flag (type-stripping, no transpile) plus a small custom ESM loader resolving the `@/` path alias — a real, reproducible execution of the real `utils/csv.ts`, `lib/request-guard.ts`, `lib/export-path.ts`, `services/copernicus-truth.ts`, `services/copernicus-engine.ts`, and a real scan of `components/**/*.tsx` against the real `config/capabilities.json`. All checks passed, including a dedicated behavioral-equivalence check confirming the WP0.4 truth-contract refactor below did not change `fetchSentinel1Metadata()`'s actual output. This process caught one real mistake before it shipped: test 2 initially assumed `sanitizeCsvValue` preserves the input type for non-triggering values (matching the Python side's behavior) — actual execution showed the TypeScript version always returns a `string`, which is documented in `03_SECURITY_REMEDIATION.md` and reflected correctly in the final test.

**Action needed on your machine:** run `npm install` (installs `vitest`), then `npm test`. That will be the first time these exact test files run through the actual framework named in this document, though the underlying logic they assert has already been verified to be correct.

## Follow-up, attempt 1 (insufficient): externalizing `node:` specifiers in `vitest.config.ts`

After `npm install` and `npm run build` succeeded on your machine, `npm test` failed with Vitest unable to resolve `node:sqlite`. Root cause as first diagnosed: `node:sqlite` is a newer, still-experimental Node built-in. `next.config.ts` already carries `serverExternalPackages: ["node:sqlite"]` to tell Next's own bundler to leave it alone — which is why the production build works — but that setting is Next.js-specific and Vitest's independent Vite-based pipeline never reads `next.config.ts`.

The fix attempted first was to declare `node:`-prefixed specifiers external in `vitest.config.ts`, via both `test.server.deps.external` and top-level `ssr.external`. **This did not work.** The exact same error persisted after the change: `Failed to load url sqlite (resolved id: sqlite). Does the file exist?` — note the specifier by the time it fails is bare `sqlite`, not `node:sqlite`. Something in this Vite/Vitest version's dependency-scan pipeline strips the `node:` prefix before the externalization check the config change targeted ever runs, so an `external: [/^node:/]` rule never gets a chance to match — it's checking the wrong (already-transformed) string. This is left in `vitest.config.ts` as harmless defense-in-depth for any future `node:`-prefixed import (`node:path` and `node:fs`, used directly by several test files, were never affected — only `node:sqlite` specifically, which is newer and evidently missing from whatever builtin-recognition list this Vite version's dependency scanner uses), but it is not, on its own, what fixed this failure.

## Follow-up, attempt 2 (decisive fix): isolating the pure truth contract from the database

Since configuring the test runner around `node:sqlite` proved unreliable, the actual fix removes the dependency instead of trying to hide it from Vite. This is the lower-risk approach: it doesn't depend on correctly predicting one specific bundler's internal builtin-detection behavior, which is exactly what went wrong with attempt 1.

**What changed:**
- **New file `services/copernicus-truth.ts`** — a module with **zero import statements of any kind** (verified by direct inspection: `grep '^import' services/copernicus-truth.ts` matches nothing). It exports the truth-contract constants (`COPERNICUS_DATA_STATUS`, `COPERNICUS_SOURCE`, `COPERNICUS_IS_REAL_SATELLITE_EVIDENCE`, `MOCK_SCENE_ID_PREFIX`, `EXPECTED_SENTINEL1_CAPABILITY_STATUS`), a `copernicusTruthMetadata()` function that builds the shared `{ dataStatus, source, isRealSatelliteEvidence }` object, and an `isTruthfulCopernicusResponse()` validator plus `allScenesAreMarkedSynthetic()` helper. Because it has no imports at all, there is no possible path from this module to `node:sqlite`.
- **`services/copernicus-engine.ts` modified** to import `copernicusTruthMetadata` and `MOCK_SCENE_ID_PREFIX` from the new module and spread/reference them at all four response-construction sites (`fetchSentinel1Metadata`'s two return paths, `copernicusForSite`, `copernicusStatus`) instead of writing the three-field triplet and the `"MOCK_S1_"` literal out by hand in each place. This is a pure refactor — verified byte-for-byte behaviorally equivalent (same fields, same values, same scene-id format) via direct execution; `services/copernicus-engine.ts` itself, `lib/db.ts`, and every other database-touching file are otherwise unchanged, and the production database is not touched by this change in any way.
- **`tests/copernicus-truth.test.ts` rewritten** to import only `@/services/copernicus-truth` — it no longer imports `@/services/copernicus-engine` at all, so it no longer has any transitive path to `@/services/site-service` → `@/lib/db` → `node:sqlite`. This is the actual mechanism that fixes the collection failure: there is nothing left in this test's import graph for Vite to mis-resolve.
- **New file `tests/copernicus-engine-contract.test.ts`** — since the truth test above no longer touches `services/copernicus-engine.ts` at all, this companion test closes that gap: it reads `copernicus-engine.ts` as plain text via `fs.readFileSync` (never importing it as a module — the same source-inspection technique `tests/capabilities-registry.test.ts` already used for `components/**/*.tsx`) and asserts, via string/regex checks, that the engine actually imports and calls `copernicusTruthMetadata()` at every response site, uses `MOCK_SCENE_ID_PREFIX` rather than a re-hardcoded prefix, and contains no reintroduced inline `dataStatus: "simulated"` / `source: "local_rule_engine"` / `isRealSatelliteEvidence: false` literals. Together, `copernicus-truth.test.ts` proves the contract itself is correct, and `copernicus-engine-contract.test.ts` proves the engine actually honors it — without either test ever loading `node:sqlite`.

**What was explicitly not done, per the low-risk requirement:** `node:sqlite` was not mocked globally; no npm package literally named `sqlite` was installed (there is no such correct package for an `import ... from "node:sqlite"` specifier); no application database behavior changed; no test assertion was weakened, skipped, or removed to work around the failure — the same truth guarantees are checked, now from a position where they can't fail to load for unrelated bundler reasons.

**Verification performed from this session:** every new/changed piece was directly executed via Node's `--experimental-strip-types` — confirmed `services/copernicus-truth.ts` has zero imports, confirmed both new test files' logic passes, confirmed the source-inspection contract check correctly passes against the real (refactored) `copernicus-engine.ts`, and confirmed `fetchSentinel1Metadata()`'s actual runtime output is unchanged by the refactor (same truth fields, same `MOCK_S1_`-prefixed scene ids, for both the valid- and invalid-coordinate code paths). A full-suite equivalent run (all 6 test files' worth of assertions, executed the same way) passed, plus an explicit check that no file under `tests/` imports `@/services/copernicus-engine` or `@/lib/db` — the specific condition that caused the original failure.

This session cannot literally run `npm test` to observe a pass. **You'll need to run `npm test` yourself** for the authoritative confirmation; the diagnosis, the fix, and the equivalence checks above are all backed by direct execution evidence, but none of that is a substitute for actually seeing Vitest itself report all suites green on your machine.
