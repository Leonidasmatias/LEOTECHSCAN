# 13 — Migration Strategy (Genesis Phase 2)

Incremental migration, no rewrite, no existing API broken (Principle 4). Stages below are sequential per capability being migrated (e.g. Data Trust goes through all six stages before Confidence starts its own), not global lockstep phases — this lets Data Trust reach "canonical endpoint addition" while Evidence is still in "shadow mode," consistent with Principle 1.

## Stages

1. **Shadow mode.** The new adapter computes its canonical output on every request the legacy engine already serves, but the result is only logged (via `06_EVENT_MODEL.md`'s events / `11_OBSERVABILITY_MODEL.md`'s logging), never returned to any caller. Zero user-visible change. Purpose: confirm the adapter runs without error against real production data volume before anyone depends on it.
2. **Dual execution.** Both legacy and canonical paths compute and are both logged, still with only the legacy result served. Purpose: accumulate a large enough sample of (legacy, canonical) pairs for stage 3's comparison to be statistically meaningful, not just "it worked once."
3. **Output comparison.** Automated comparison of the legacy and canonical results for the same input, against a **documented tolerance threshold** (e.g. for Data Trust: canonical `Score.value * 100` must be within ±2 of legacy `trustScore` for ≥99% of sampled sites, given the conversion is meant to be mechanical per `02_CANONICAL_DOMAIN_MODEL.md`'s scale decision — a larger discrepancy indicates an adapter bug, not an acceptable rounding difference). Comparison failures block progression to stage 4; they do not silently pass. **Clarified by Increment 6.5 (`23_INCREMENT_6_5_ARCHITECTURAL_DECISIONS.md`, Decision D):** "block progression to stage 4" means block *activation* of stage 4 — i.e. no existing caller may be switched onto the canonical path, and the canonical path may not become anyone's default, until stage 3 has passed. It does not mean the stage-4 route's mere existence is forbidden before stage 3 completes; see stage 4's own clarification below.
4. **Canonical endpoint addition.** A new route (e.g. `GET /api/intelligence/data-trust/site`) is added, additive only — the legacy route is untouched (Principle 4). The new route is the first place a canonical result is actually served to a real caller. **Clarified by Increment 6.5, Decision D:** an isolated, additive, authenticated, read-only instance of this route may be added — and its own increment (Increment 7) completed — before stage 3's comparison work (Increment 8) runs, **provided** the route has zero migrated callers, is not referenced by any existing UI/route/consumer, is not marked as a replacement for the legacy route, and is not treated as the default path for anything. Stage 3 still gates every subsequent stage that involves a real caller depending on the canonical result (stage 5 UI migration onward) — it does not gate the route merely existing and being independently callable/testable. This reflects the reality that "add an inert, additive, opt-in endpoint" and "make something depend on that endpoint" are different risk levels, and only the second is what stage 3's comparison evidence is protecting against.
5. **UI migration.** The frontend (`components/**`) is switched, feature-flagged, to call the new canonical endpoint for a specific feature — behind a flag so it can be reverted instantly if a real-world discrepancy surfaces that stage 3's sampling missed.
6. **Legacy route deprecation.** Only after stage 5 has run in production for the documented coexistence duration (below) with no flag-triggered rollback, the legacy route is marked `deprecated` in its manifest/documentation (not removed) and `config/capabilities.json` is updated to point at the canonical capability.
7. **Legacy engine retirement.** Only after the deprecated route has measurably zero traffic for a documented period — the legacy engine file itself is only then a candidate for removal, and removal is its own separately-approved, separately-reviewed change, not bundled into any of the stages above.

## Definitions

- **Compatibility strategy:** additive-only routes (Principle 4) — no existing route's request/response contract changes at any stage until stage 6, and even stage 6 only *deprecates*, does not remove.
- **Feature flags:** a simple boolean per capability (e.g. `USE_CANONICAL_DATA_TRUST`), read from environment/config — no feature-flag *platform* is justified at this scale (Principle 17); a plain config value read at request time is sufficient and matches the codebase's existing config-file-driven style (`config/*.json`).
- **Capability registry updates:** `config/capabilities.json` is updated at stage 6 (deprecation), not earlier — during stages 1–5 the capability's public-facing status is unaffected, since nothing user-visible has changed yet (Principle 16: don't advertise before true, in either direction — a capability isn't "more operational" just because a shadow computation exists).
- **Shadow-result comparison:** per stage 3, automated, sampled (not necessarily exhaustive — full-dataset comparison for every request is disproportionate; a statistically adequate sample, size TBD per capability's risk level, is sufficient).
- **Tolerance thresholds:** engine-specific, documented alongside the engine's own migration record (not a single global number) — Trust's ±2-points-on-100-scale example above is illustrative, not universal; a binary classification (e.g. Coordinate Assessment's `status` enum) would use exact-match tolerance instead, by nature.
- **Rollback:** at any stage 1–5, rollback is instant (flip the feature flag / stop reading the shadow log) since no user-facing behavior depends on the canonical path yet, or depends on it only behind an instantly-revertible flag.
- **Deprecation criteria:** stage 5's coexistence duration elapsed + zero flag-triggered rollbacks during that period + stage 3's comparison passing continuously (not just once) during dual execution.
- **Data migration criteria:** N/A for Phase 2's adapter work in the general case (adapters read existing tables, they don't move data) — would only apply if a future increment introduces a new persisted table that supersedes an old one (`09_PERSISTENCE_AND_HISTORY.md`), and is out of scope until that's proposed.
- **Coexistence duration:** not fixed by this document as a single global number (e.g. "30 days") — recommend it be set per-capability at the time stage 4 begins, based on that capability's actual traffic pattern (a low-traffic endpoint needs a longer window to accumulate a meaningful zero-rollback track record than a high-traffic one). Flagged as an open operational decision, not architecturally pre-committed.
- **Proof required before switching traffic (stage 4→5):** stage 3's comparison must have passed continuously through at least one full dual-execution cycle covering a representative traffic sample (not a single manual test) — "it worked when I tried it" is explicitly insufficient per this document's standard.

## How this fixes the two audit-flagged problems without breaking anything

- **GET-with-side-effects** (`/api/data-trust/site`, `/api/evidence-center/site`): the fix is **not** "silently stop persisting on the existing route" (that would be an undocumented behavior change, violating Principle 4/5). The fix runs through this exact strategy: the new canonical route (stage 4) is genuinely read-only from day one (Principle 6); the *legacy* route's side effect is only addressed at a stage 6/7-adjacent, separately-approved step once traffic has moved to the canonical route and the legacy route's behavior can be safely changed without surprising an existing caller — or, if that's judged too slow given the security severity, as an **explicitly separate, expedited, Increment-0-adjacent fix** called out on its own in `14_IMPLEMENTATION_ROADMAP.md`, not smuggled into this migration's normal cadence.
- **Missing authentication:** orthogonal to this migration strategy entirely — auth is a cross-cutting boundary (`10_SECURITY_BOUNDARY.md`) applied uniformly once implemented, not something that migrates capability-by-capability.
