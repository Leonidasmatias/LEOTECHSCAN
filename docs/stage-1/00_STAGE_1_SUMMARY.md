# Stage 1 — Geospatial Intelligence Foundation — Summary

Living summary, updated as each checkpoint closes. See `docs/stage-0/*.md` for the Stage 0 baseline this builds on (commit `0b61fd7`), and `docs/audit-v4/*.md` for the original audit this whole program answers to.

Scope for this stage: a geospatial domain model, coordinate-quality classification, a spatial index, a deterministic national grid, and the read-side spatial intelligence service that will back later viewport/cluster/radius APIs and map UI work (Checkpoints 3–5). **No Copernicus/Sentinel-1 real integration, no Stage 2 work, is in scope or has been started.**

### Checkpoint 2 — Spatial index, national grid, spatial intelligence engine (complete)

- **WP1.4 Spatial Index**: `scripts/geospatial-spatial-index.mjs` builds `site_spatial_index`, an SQLite R-Tree virtual table (`CREATE VIRTUAL TABLE ... USING rtree`), verified available in Node's built-in `node:sqlite` (the same engine `lib/db.ts` uses at runtime) with an automatic fallback to a plain composite index on `sites(latitude, longitude)` if R-Tree isn't available in a given SQLite build. Validated on a disposable database copy first (dry-run → build → verify → measured performance), then applied to the live database. See `04_SPATIAL_INDEX.md` for full measurements and the rebuild/rollback procedure.
- **WP1.5 National Grid**: `services/geospatial/national-grid.ts` — a deterministic fixed-cell grid (4 resolutions, ~111km down to ~111m), not geohash or H3 (see `05_NATIONAL_GRID.md` for why). Pure, dependency-free, unit-tested (8 tests).
- **WP1.6 Spatial Intelligence Engine**: `services/geospatial/spatial-intelligence-engine.ts` implements `getSitesInBoundingBox`, `getSiteCountInBoundingBox`, `getClustersInBoundingBox`, `getSitesWithinRadius`, `getNearestSites`, `getGridSummary`, `getCoordinateQualitySummary`, `getSiteGeospatialStatus` — every one of them limit-capped, none can serialize the full national dataset. Verified against a real in-memory `node:sqlite` database (not mocks) exercising the actual R-Tree index, including a point deliberately placed outside Brazil to confirm bounding-box queries never leak it in. `rebuildSpatialIndex` and `evaluateGeospatialEligibility` (batch classification writing to `site_geospatial_status`) are deferred to Checkpoint 4's WP1.11 processing-command work, where they belong alongside the other CLI commands, rather than being wired in twice.

### Checkpoint 3 — Geospatial APIs (complete, pending local validation)

- **WP1.7-1.10**: four read-only endpoints — `GET /api/geospatial/viewport`, `/clusters`, `/radius`, `/nearest`, `/summary` — every one a thin adapter over the Checkpoint 2 engine (requirement 1: no query-shaping logic reimplemented). New pure modules `services/geospatial/request-params.ts` (strict validation, requirement 2) and `services/geospatial/compact-site.ts` (compact response shaping, requirement 5) are shared across all five routes. Every result count is hard-capped inside the engine regardless of request params (requirement 3); bounding-box/radius queries are always prefiltered through the WP1.4 R-Tree spatial index (requirement 4); grid clustering is deterministic (requirement 6, unchanged from Checkpoint 2's `aggregateIntoGridClusters`). Full design, request/response contract, and validation rules: `07_GEOSPATIAL_APIS.md`.
- **Checkpoint 3 finding, fixed in-scope**: honest performance measurement against a disposable copy of the real 299,308-row database (requirement 8) surfaced two real issues in Checkpoint 2's already-approved `getClustersInBoundingBox` — a whole-country cluster request silently sampled only ~1.7% of the national dataset (reusing the individual-site fetch's 5,000-row cap), and raising that sample size immediately hit a real SQLite "too many SQL variables" ceiling. Both fixed: a lean coordinates-only candidate fetch raises the effective cluster sample to 50,000 (~17% of the national dataset, a ~29x improvement), and a new `chunkArray()` helper batches every `id IN (...)` query at a safe 900-parameter ceiling (portable across SQLite builds going back before the 999-variable pre-3.32.0 default), applied to both the new clustering fetch and Checkpoint 2's existing `sitesByIds()`. No other Checkpoint 2 function changed. Full account: `07_GEOSPATIAL_APIS.md`.
- **Tests added**: `tests/geospatial-request-params.test.ts` (20 checks), `tests/geospatial-compact-site.test.ts` (6 checks), `tests/geospatial-api-contract.test.ts` (45 source-inspection checks across the 5 routes, requirement 7), plus a new `chunkArray` block in `tests/geospatial-spatial-intelligence-engine.test.ts` (5 checks) and one new assertion in the existing `tests/geospatial-spatial-engine-contract.test.ts`.
- **Local validation pending**: this session verified all of the above via direct execution — the pure modules and contract-test regexes dry-run against real source (mirroring the Checkpoint 2 technique), and, new for this checkpoint, the actual engine functions run directly against a disposable, SHA-256-verified, integrity-checked copy of the live database (8 correctness checks + the performance table in `07_GEOSPATIAL_APIS.md`), all passing. The route files' literal HTTP layer could not be executed end-to-end in this session (this project's `node_modules`/`next` exist only on the Windows host; the device bridge's own execution path hit a known, previously-documented bridge read-cache issue trying to resolve it, not a real code problem) — covered instead by the source-inspection contract test. Checkpoint 4 does not begin until the user confirms `npm test` passes fully, `npx tsc --noEmit` passes, and Stage 0 regression tests remain green.
