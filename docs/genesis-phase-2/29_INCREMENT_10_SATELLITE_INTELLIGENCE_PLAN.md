# 29 — Increment 10: Satellite Intelligence — Architecture Audit and Implementation Plan

Planning-only document. No production code, test, or configuration file is
created or modified by this document.

## 1. Executive summary

Increment 10 introduces **Satellite Intelligence** as a new, fully
independent canonical capability — a peer to Data Trust (Increment 7) and
Evidence Center (Increment 8), not an extension of either, and explicitly
**not** wired into the frozen Site Intelligence Aggregator (Increment 9)
in this increment. It composes a provider-neutral satellite observation
port, whose sole Increment 10 implementation wraps the repository's
existing, already-frozen-in-spirit legacy Copernicus/Sentinel-1 subsystem
(`services/copernicus-engine.ts`, `services/copernicus-truth.ts`,
`config/copernicus_rules.json`) — which is **entirely simulated data**,
a binding, repository-wide truth invariant this plan inherits and never
weakens. The capability is metadata-first: no imagery bytes, no binary
persistence, no signed URLs. It publishes its own canonical `Evidence`
records for future consumers but does not feed Data Trust and does not
create any circular dependency. The safest, smallest architecture that
preserves future extensibility is selected throughout, consistent with
every prior increment's own discipline.

## 3. Current architecture audit

### 3.1 Correction to the mission's own assumed file list

The mission's Step 3 lists `services/intelligence-models/**`,
`services/intelligence-providers/**`, and `services/evidence-center/**`
as existing directories to inspect. **None of these three exist in this
repository** — confirmed by direct directory listing, not inferred. The
real, relevant structure is `services/intelligence/**` (generic canonical
contracts, frozen since Genesis Phase 1), `services/intelligence-adapters/**`
and `services/intelligence-runtime/**` (Genesis Phase 2's own layers),
and `services/evidence-center-engine.ts` (a single legacy file, not a
directory). This correction matters: it means Increment 10 is not
resuming or extending some existing provider/model layer — it is
introducing the first one, exactly as Increments 7/8/9 each introduced
their own first-of-a-kind layer before it.

### 3.2 The pre-existing legacy satellite subsystem (critical finding)

Direct reading of `services/copernicus-truth.ts`, `services/copernicus-engine.ts`,
`services/satellite-validation-engine.ts`, and `config/copernicus_rules.json`
uncovered a fully-formed, already-mature legacy satellite subsystem this
plan must build on, not duplicate:

- **`COPERNICUS_IS_REAL_SATELLITE_EVIDENCE = false`** (`copernicus-truth.ts`)
  — a binding, dependency-free, already-unit-tested truth invariant:
  *there is no real HTTP client to the Copernicus Data Space Ecosystem
  anywhere in this codebase.* Every scene is synthetic
  (`MOCK_S1_`-prefixed scene IDs), generated by a deterministic local
  rule engine. A documented "STAGE 0 TRUTH CORRECTION" comment records
  a past incident where configured credentials created a false
  impression of real integration, and explicitly forbids ever
  reintroducing a credential-conditioned branch until a real Stage 3
  client exists. **Increment 10 inherits this invariant unconditionally**
  — nothing it produces may claim real satellite evidence.
- **`copernicusForSite(db, siteId, radiusKm, lookbackDays, persist)`**
  already has the exact `persist=false`-capable shape Increments 7/8's
  own outer read adapters rely on, already unconditionally calls
  `ensureCopernicusTables(db)` first (the same "logically read-only for
  business data, technically capable of a one-time schema write"
  classification already established twice), and already returns rich
  per-scene metadata (`acquisitionDate`, `orbitDirection`, `polarization`,
  `relativeOrbit`, `sceneId`, `cloudNote`, `sourceUrl`, `provider`,
  `mission`, `productType`) plus a deterministic 0–100
  `validation.validationScore`/`evidenceLevel` derived from
  `config/copernicus_rules.json`'s own scoring rules (recency threshold:
  45 days, per `rules.scoring.recentSceneDays`).
- **A trap, found only by reading the actual implementation, not the
  filename:** `services/satellite-validation-engine.ts`'s
  `satelliteValidationForSite()` — the function `confidence-engine.ts`'s
  own legacy chain already calls — internally calls
  `copernicusForSite(db, siteId, undefined, undefined, true)` **when no
  cached validation row exists yet**, hardcoding `persist=true`. Any
  Increment 10 outer adapter must call `copernicusForSite` **directly**,
  never `satelliteValidationForSite`, or it will silently inherit an
  unintended persistence path.
- **Evidence Center's existing `COPERNICUS` evidence item is thin, not
  rich.** Direct reading of `services/evidence-center-engine.ts` (line
  22) shows it constructs exactly one flat item:
  `{type: "COPERNICUS", source: "Sentinel-1 metadata_only", status: copernicus?.validation.evidenceLevel, summary: copernicus?.recommendation}`
  — discarding every per-scene field entirely. Increment 8's frozen
  Evidence Adapter (`services/intelligence-adapters/evidence-adapter.ts`)
  already has full, hardcoded, tested support for translating this one
  thin item, including a mandatory `copernicus_evidence_simulated` issue
  and a forced-low `reliability = 0.1` via `copernicusTruthMetadata()`
  reuse — but it is structurally incapable of representing multiple
  scenes or any of the richer metadata this increment's mission requires.
  Increment 10 must **not** attempt to extend or reuse this adapter for
  its own, richer evidence (see Section 8).
- **`services/geospatial/coordinate-quality-engine.ts`'s `evaluateCoordinateQuality`**
  already computes an `eligibleForSentinel: boolean` field, whose own
  header comment states its exact purpose: *"the coordinate is precise
  and unambiguous enough to be worth using in a future Sentinel-1 scene
  search... NOT a claim that any satellite evidence exists."* This is a
  pure, dependency-free, already-frozen-in-spirit module purpose-built
  for exactly the "resolving site coordinates" responsibility this
  increment's mission assigns to the new capability. It must be reused
  verbatim, never reimplemented.
- **The legacy route `app/api/copernicus/site/route.ts`** already calls
  `copernicusForSite(getWritableDb(), id, undefined, undefined, false)`
  — unauthenticated (predates Increment 0), unmodified, remains
  untouched by this plan.

## 5. Scope and non-scope

### In scope
- Validating a Satellite Intelligence request (`id` parameter, same convention as every other capability).
- Resolving site coordinates from the canonical Site Entity Adapter (Increment 3, reused verbatim) and gating on `evaluateCoordinateQuality`'s `eligibleForSentinel` (Section 3.2).
- Requesting satellite observations through a provider-neutral port, whose sole Increment 10 implementation wraps the legacy Copernicus engine.
- Adapting provider-specific metadata into normalized, canonical satellite observations.
- Computing deterministic quality/freshness indicators, reusing the legacy engine's own existing scoring rules (no new business formulas).
- Attaching evidence and provenance via a new, dedicated satellite evidence adapter.
- Returning an explicit capability state (`available`/`unavailable`/`notFound`, mirroring Increment 9's own vocabulary) and sanitized operational diagnostics.

### Explicitly out of scope
- Computer-vision inference of any kind (no frozen CV capability exists in this repository to wrap; inventing one would violate the adapter-purity "no new business formulas" rule and is a distinct, much larger increment).
- Predictive risk scoring or environmental risk modeling.
- Street-level imagery (a conceptually separate future capability).
- User-facing map rendering (a UI concern, not this capability's).
- Writing or mutating any site record.
- Authentication (reuses `requireAdminAuth` verbatim, no new mechanism).
- Direct provider secrets exposure of any kind.
- Generating unsupported certainty — every observation this capability emits must be traceable to a deterministic, disclosed source computation, never a visual judgment call.
- Changing frozen Data Trust semantics or formula.
- Modifying the Increment 9 public envelope, orchestrator, or aggregator.
- A real (non-simulated) satellite provider integration — the port is designed to support one later; building one is not this increment's job (no API keys, network client, or cost approval exist for it today).

## 6. Selected architecture

**A new, fully independent canonical capability**, structurally identical
in shape to Data Trust (Increment 7) and Evidence Center (Increment 8) at
the *file-organization* level: outer read adapter → pure orchestrator
core → orchestrator instance → pure projection adapter → route/handler
split with auth-before-dynamic-import. It differs from Data Trust/Evidence
Center at the *execution-model* level: **the provider port, the
orchestrator core, and the orchestrator instance are asynchronous
(`Promise`-returning) from Increment 10 onward** — see Section 10's F-3
hardening for the full rationale. This is a deliberate, disclosed
divergence from Increment 7/8/9's synchronous convention, not an
oversight; Section 14 states its consequence for future integration
explicitly.

The capability composes a **provider port** (a new abstraction this
increment introduces, since no prior increment needed one) whose sole
concrete implementation wraps the legacy Copernicus engine. The port's
own types are genuinely provider-neutral — no Copernicus/Sentinel-shaped
type crosses it (Section 10).

```
GET /api/intelligence/satellite/site?id=<siteId>
  → requireAdminAuth (checked first, before any dynamic import)
  → (only if authorized) dynamic import of:
      services/intelligence-runtime/satellite-intelligence-orchestrator-instance.ts
        → services/intelligence-adapters/io/satellite-site-read-adapter.ts   (DB-touching, site row only)
        → services/intelligence-adapters/io/legacy-copernicus-provider.ts    (DB-touching, implements the provider port, persist=false literal)
        → services/intelligence-adapters/site-entity-adapter.ts         (Increment 3, reused verbatim)
        → services/geospatial/coordinate-quality-engine.ts              (reused verbatim)
        → services/intelligence-adapters/satellite-observation-adapter.ts (pure, new)
        → services/intelligence-adapters/satellite-evidence-adapter.ts   (pure, new)
        → services/intelligence-runtime/satellite-intelligence-provider-port.ts (pure interface, new -- see Section 10 for why this lives in intelligence-runtime/, not intelligence-adapters/)
        → services/intelligence-runtime/satellite-intelligence-orchestrator.ts (pure core, async)
  → app/api/intelligence/satellite/site/handler.ts (async, awaits the orchestrator)
      → services/intelligence-adapters/satellite-projection-adapter.ts (pure, new, synchronous -- projection itself has no I/O and needs no awaiting)
```

The two new DB-touching outer files —
`services/intelligence-adapters/io/satellite-site-read-adapter.ts` and
`services/intelligence-adapters/io/legacy-copernicus-provider.ts`

require modifying frozen files and their own exclusion-list entries for
no functional benefit; that relocation, if ever wanted, is its own
separately-scoped future refactor, not part of Increment 10.

## 7. Rejected alternatives

- **Extending `/api/intelligence/site` (Site Intelligence) directly in
  Increment 10.** Rejected: touches the frozen Increment 9 envelope and
  orchestrator against Increment 9's own explicit "not now" signal;
  couples two independently-versioned capabilities' release cadence;
  provides no benefit a standalone route doesn't already provide. A
  future aggregator-extension increment can still wire in Satellite
  Intelligence's field-shape-compatible entry-point function, though
  (post-review hardening, F-3) that future increment will need to be
  explicitly authorized to perform an asynchronous composition change (or
  introduce a precomputed-result boundary), since Satellite Intelligence's
  own entry point is `Promise`-returning while Increment 9's aggregator
  is synchronous — not a zero-cost wire-in, but still no reason to do that
  work prematurely, inside Increment 10, before it is needed.
- **Reusing the existing Evidence Adapter for satellite evidence** (via
  synthetic `LegacyEvidenceItem`-shaped objects with a new `type`
  literal). Rejected: technically possible (the adapter explicitly
  tolerates an unrecognized `type` as a non-blocking, disclosed
  fallback), but the narrow four-field `LegacyEvidenceItem` shape cannot
  losslessly carry per-scene metadata (acquisition date, orbit,
  polarization, scene id, cloud note) without concatenating it into a
  single opaque `summary` string — a real information loss this
  increment's own canonical-model ambition (Section 9) cannot accept.
- **A commercial high-resolution provider as the mandatory Increment 10
  provider.** Rejected: no such integration, API key, cost approval, or
  network client exists in this repository today; mandating one would
  turn a planning-and-architecture increment into an unapproved
  procurement and infrastructure decision. The provider port is designed
  to support one later without redesign.
- **Persisting fetched imagery metadata to new tables.** Rejected: no
  business need identified; every prior increment's own "read-only until
  proven otherwise" discipline applies equally here; the legacy
  `copernicus_scenes`/`site_satellite_validation` tables already persist
  this exact data when the legacy routes are used with `persist=true`,
  unchanged.
- **Retrieving or processing imagery bytes.** Rejected — see Section 11.
- **A dedicated `services/intelligence-satellite/` top-level directory**
  (mirroring neither `intelligence-adapters/` nor `intelligence-runtime/`).
  Rejected as an unjustified new top-level convention when the existing
  two-directory split (adapters vs. runtime), plus the new `io/`
  subdirectory, already accommodates every file this increment needs
  without inventing a third top-level intelligence directory.
- **Option D (post-review hardening, F-8) — a thin direct wrapper over
  the legacy Copernicus engine, with no provider-port abstraction at
  all.** Genuinely considered, not dismissed by omission this time.
  Advantages: smaller initial file count; less abstraction to design,
  build, and keep correct; mirrors Data Trust/Evidence Center's own
  minimal wrapping exactly, with no new architectural concept at all.
  Disadvantages, and the reason this plan still rejects it: tighter
  legacy coupling with no clean seam for a future real provider — any
  eventual real-provider work would have to retrofit an abstraction into
  code that was never designed to have one, a strictly more expensive
  path than introducing the (now-corrected, genuinely neutral — Section
  10.2) seam now; Copernicus/SAR-specific types would leak all the way
  into the canonical model with no natural boundary stopping them (the
  independent review's own F-2 finding demonstrates how easily this
  leakage happens even *with* an explicit port to guard against it — a
  design with no port at all has no such guard whatsoever); testing
  provider failure modes (timeout, misconfiguration, rate limiting) would
  be far less isolated, since there would be no seam at which to inject a
  deterministic fake. The corrected port (Section 10) is deliberately
  minimal — an interface plus one implementation, not a speculative
  multi-provider framework — which keeps this rejection's own cost
  honest: the abstraction earns its keep specifically because it is thin,
  not because "more abstraction is always better."
- **Option E (post-review hardening, F-8) — retire or replace the legacy
  Copernicus subsystem before introducing Satellite Intelligence.**
  Rejected, and not a close call: `services/copernicus-engine.ts` has
  multiple live consumers today (`services/evidence-center-engine.ts`,
  `services/confidence-engine.ts` via `satellite-validation-engine.ts`,
  and the legacy `app/api/copernicus/**` routes) — retiring it is an
  unrelated, much larger migration project governed by
  `13_MIGRATION_STRATEGY.md`'s own seven-stage process, with real
  regression risk to systems this increment has no mandate to touch.
  There is no need to retire the legacy subsystem in order to isolate a
  new canonical capability from it — that isolation is exactly what the
  outer-adapter/port pattern (Section 6, Section 10) already provides,
  the same way Increment 7 isolated Data Trust from
  `services/data-trust-engine.ts` without retiring it. Future
  deprecation of the legacy Copernicus subsystem, if it ever happens,
  can only follow once a canonical replacement exists *and* migration
  evidence justifies it — exactly the same order every other legacy
  subsystem in this project is expected to follow, per Principle 4 and
  the migration strategy's own explicit staging.

## 9. Canonical data model

All types are **capability-specific**, living in
`services/intelligence-adapters/satellite-observation-model.ts` — not
added to the generic, frozen `services/intelligence/**` contract layer,
exactly mirroring how Increment 9's own `SiteIntelligenceStatus`/etc. are
capability-specific types in `intelligence-runtime/`, not additions to
`services/intelligence/`. `services/intelligence/**` remains completely
untouched by this increment, as it has been since Genesis Phase 1.

```ts
// Truth metadata (post-review hardening, F-1) -- MANDATORY, non-optional,
// carried once at the result/envelope level (Section 14/15), never
// omittable, never inferred from a provider name string alone. See
// Section 10 for the exact derivation rule and the binding requirement
// that only an adapter which can prove a real provider response may ever
// report "provider_sourced".
export interface SatelliteTruthMetadata {
  readonly dataReality: "simulated" | "provider_sourced";
  readonly realSatelliteEvidence: boolean;
  readonly simulationReason: string | null; // non-null whenever dataReality === "simulated"
  readonly sourceDisclosure: string;        // human-readable, e.g. "dataStatus=simulated; source=local_rule_engine"
}

// Provider identity
export interface SatelliteProviderIdentity {
  readonly providerCode: string;        // e.g. "copernicus-legacy-simulated"
  readonly dataset: string;             // e.g. "Sentinel-1 GRD"
  readonly sourceType: "sar" | "optical" | "unknown";
  readonly retrievedAt: string;         // ISO-8601, injected clock, not Date.now()
}

// Spatial metadata
export interface SatelliteSpatialMetadata {
  readonly siteCoordinates: { readonly latitude: number; readonly longitude: number };
  readonly requestedRadiusKm: number;
  readonly coordinateEligibility: "eligible" | "ineligible"; // derived from evaluateCoordinateQuality.eligibleForSentinel
  readonly coordinateStatus: string;    // CoordinateStatus, passed through verbatim
}

// Temporal metadata
export interface SatelliteTemporalMetadata {
  readonly captureTime: string | null;  // scene's own capturedAt; null only if the provider did not report one (never fabricated)
  readonly retrievedAt: string;
  readonly imageryAgeDays: number | null; // deterministic: (retrievedAt - captureTime) in days; null when captureTime is null
  readonly freshness: "recent" | "stale" | "unknown"; // "unknown" only when imageryAgeDays is null; threshold reused from config/copernicus_rules.json's recentSceneDays
}

// Quality metadata (post-review hardening, F-2: provider-neutral --
// nullable numeric fields replace the earlier "not_applicable"/"unknown"
// enum, which was itself a mild Copernicus/SAR-specific assumption baked
// into a canonical type. A null value honestly means "this source did
// not report this metric" -- true for the current SAR-based legacy
// provider's cloud coverage, and equally usable, without a special case,
// for any future provider that also doesn't report a given metric.)
export interface SatelliteQualityMetadata {
  readonly cloudCoveragePercent: number | null;   // 0-100, or null if not reported by this source
  readonly noDataCoveragePercent: number | null;  // 0-100, or null if not reported
  readonly spatialResolutionMeters: number | null; // or null if not reported (the legacy provider does not report this today)
  readonly sourceConfidence: number;    // 0-1, already normalized by the adapter -- never a raw legacy 0-100 value
  readonly qualityClassification: "high" | "medium" | "low" | "insufficient"; // mapped from legacy evidenceLevel ("Alta"/"Media"/"Baixa"/"Insuficiente")
  readonly usable: boolean;             // qualityClassification !== "insufficient"
}

// One normalized observation (one provider scene → one observation)
export interface SatelliteObservation {
  readonly observationId: string;       // deterministic, derived from siteId + sourceSceneId (Section 10, F-4)
  readonly observationType: "sar_scene_metadata"; // the only type Increment 10 produces; new types are additive later
  readonly provider: SatelliteProviderIdentity;
  readonly spatial: SatelliteSpatialMetadata;
  readonly temporal: SatelliteTemporalMetadata;
  readonly quality: SatelliteQualityMetadata;
  readonly derivationMethod: string;    // e.g. "legacy-copernicus-provider:v1" -- versioned, not a free-text guess
  readonly evidenceId: string;          // EvidenceId of the satellite-evidence-adapter's own Evidence record for this observation -- exact formula in Section 10 (F-4)
  readonly limitations: readonly string[]; // e.g. "Simulated data -- no real satellite integration exists yet." -- see the mandatory truthMetadata field (Section 14/15) for the structural guarantee; this array remains free-form, human-readable, supplementary disclosure, never the sole truth signal
}

// Capability/aggregate state vocabulary (post-review hardening, F-5:
// "unavailable" is now its own aggregate status, distinct from "failed" --
// see Section 17 for the full, deterministic derivation rule)
export type SatelliteCapabilityState = "available" | "unavailable" | "notFound";
export type SatelliteAggregateStatus = "complete" | "partial" | "unavailable" | "failed" | "notFound";
```

**Deliberately not fabricated:** no observation type beyond
`"sar_scene_metadata"` is defined — the mission's own instruction ("Do
not fabricate observation types merely to populate the design") is
honored by deriving exactly one type from what the legacy engine actually
produces (SAR scene metadata), leaving the union open for additive future
types (e.g. `"optical_scene_metadata"`) only when a real second source
exists to justify one.

**Source facts vs. derived facts vs. confidence vs. issues vs. evidence**
are kept structurally separate: `provider`/`spatial.siteCoordinates`/
`temporal.captureTime` are source facts (verbatim from the provider
scene); `temporal.freshness`/`quality.qualityClassification`/`quality.usable`
are derived facts (deterministic functions of source facts);
`quality.sourceConfidence` is confidence (a number, never conflated with a
boolean determination); issues live in the orchestrator's own issue list
(Section 17-equivalent pattern, mirroring Increments 7–9); evidence is a
separate, referenced `Evidence` record (via `evidenceId`), never embedded
inline.

**Truth metadata is mandatory, not optional, and structurally guaranteed
(post-review hardening, F-1).** `SatelliteTruthMetadata` is carried once,
at the `SatelliteCapabilityOutcome`/public-envelope level (Section
14/15) — never an optional field, never inferable only from a
`providerCode` string a consumer might not parse. **The current legacy
Copernicus/Sentinel engine is a deterministic simulated provider and must
never be presented as real satellite evidence.** Every response this
increment's sole provider implementation ever produces carries
`dataReality: "simulated"` and `realSatelliteEvidence: false`,
unconditionally, derived directly from `services/copernicus-truth.ts`'s
own `copernicusTruthMetadata()` — never inferred, never hand-rolled a
second time. A provider adapter may only ever report `"provider_sourced"`
when it can affirmatively prove the data it received came from a real
provider response (a binding requirement recorded here for any future
provider, since Increment 10's own sole provider can never satisfy it).
This guarantee must survive projection unchanged (Section 15) and is
test-protected (Section 24).

## 10. Provider abstraction

### 10.1 Port location (post-review hardening, F-8)

The pure port **interface** lives at
`services/intelligence-runtime/satellite-intelligence-provider-port.ts`
— `intelligence-runtime/`, not `intelligence-adapters/` — adopting the
hardening review's own recommended refinement. Rationale: the port is a
*contract the orchestrator core depends on*, structurally the same kind
of thing as `SiteIntelligenceAggregatorDeps`'s own injected-function
shape in Increment 9's orchestrator — it belongs beside the orchestrator
it serves, not among the translation adapters. This also sidesteps any
question of whether the port itself needs to live under the new `io/`
subdirectory (Section 17): it has zero I/O of its own, so it doesn't need
to, and keeping it in `intelligence-runtime/` makes that structurally
obvious rather than requiring a reader to infer it. `services/intelligence-adapters/io/`
remains reserved exclusively for I/O-touching implementations of ports
and legacy wrappers (Section 17).

### 10.2 Genuinely provider-neutral types (post-review hardening, F-2)

The independent review found the originally-proposed port embedded
undefined `LegacySatelliteScene`/`LegacySatelliteValidation` types
directly in its own "success" outcome — types that evidently aliased the
legacy engine's own Copernicus/SAR-shaped `CopernicusScene`
(`orbitDirection`, `polarization`, `relativeOrbit` are SAR-specific
fields with no meaning for, say, an optical provider). That was a real
defect: a port whose own success type is shaped around one provider's
data cannot honestly claim to be neutral, and any future, genuinely
different provider would have been forced to fabricate or awkwardly map
into a SAR-specific shape just to satisfy the port's own type. Corrected
below: the port's outcome type is built entirely from generic,
provider-agnostic fields; anything provider-specific is confined to a
bounded, sanitized `sourceAttributes` bag that never gains first-class
canonical status.

```ts
// services/intelligence-runtime/satellite-intelligence-provider-port.ts
// (pure -- an interface and its request/outcome types only, no
// implementation, no I/O, no Copernicus/Sentinel-shaped type anywhere
// in this file)

export interface SatelliteCoverageMetadata {
  readonly footprintDescription: string | null;
  readonly radiusKm: number;
}

// One provider-neutral scene/observation candidate, already reshaped by
// the adapter that produced it -- never a passthrough of a provider's
// own native response shape.
export interface SatelliteProviderScene {
  readonly sourceSceneId: string | null; // null triggers the deterministic missing-id handling in Section 10.4 (F-4)
  readonly capturedAt: string | null;
  readonly publishedAt: string | null;
  readonly retrievedAt: string;
  readonly spatialResolutionMeters: number | null;
  readonly cloudCoveragePercent: number | null;
  readonly noDataCoveragePercent: number | null;
  readonly coverage: SatelliteCoverageMetadata | null;
  // Bounded, sanitized, provider-specific preservation for provenance
  // only (e.g. the legacy adapter packs orbitDirection/polarization/
  // relativeOrbit/productType here) -- values only, never functions,
  // never raw provider response objects, never credentials or URLs
  // bearing a token.
  readonly sourceAttributes: Readonly<Record<string, string | number | boolean | null>>;
}

// Already normalized by the adapter (0-1 scale, canonical
// classification) -- the port itself never sees or forwards a raw
// legacy 0-100 score. See Section 10.4 for the exact conversion.
export interface SatelliteProviderQualitySummary {
  readonly overallScore: number; // 0-1
  readonly overallClassification: "high" | "medium" | "low" | "insufficient";
}

export interface SatelliteProviderRequest {
  readonly siteId: number;
  readonly latitude: number;
  readonly longitude: number;
  readonly radiusKm: number;
  readonly temporalWindow: SatelliteTemporalWindow; // see Section 10.5 (F-6) -- replaces the earlier, provider-shaped `lookbackDays`
}

export type SatelliteProviderOutcome =
  | { readonly kind: "success"; readonly scenes: readonly SatelliteProviderScene[]; readonly qualitySummary: SatelliteProviderQualitySummary }
  | { readonly kind: "unavailable"; readonly reason: "misconfigured" | "timeout" | "rate_limited" | "invalid_credentials" | "unexpected_error" }
  | { readonly kind: "no_coverage" };

// Post-review hardening, F-3: asynchronous from Increment 10 onward --
// see Section 10.3 for the full rationale.
export interface SatelliteProviderPort {
  readonly providerCode: string;
  fetch(request: SatelliteProviderRequest): Promise<SatelliteProviderOutcome>;
}
```

**Explicit dependency-direction rules, stated as binding requirements**:

- Provider-native payloads (whatever shape `copernicusForSite`, or any
  future provider's own client, actually returns) never cross the port —
  they are fully reshaped into `SatelliteProviderScene`/
  `SatelliteProviderQualitySummary` *inside* the adapter, before the
  `fetch()` call returns.
- Provider-native error types never cross the port — only the closed
  `unavailable.reason` union does.
- The port returns only canonical provider outcomes (the three-variant
  union above), never a provider-specific response object.
- The legacy adapter (`legacy-copernicus-provider.ts`, Section 10.3)
  converts `CopernicusScene`/`calculateSatelliteValidationScore`'s own
  output into `SatelliteProviderScene`/`SatelliteProviderQualitySummary`
  — this conversion, including the 0–100→0–1 score normalization
  (matching the established `canonical = legacy / 100` precedent), is
  entirely the adapter's responsibility, never the port's or the
  orchestrator's.
- The orchestrator core (`satellite-intelligence-orchestrator.ts`) and
  the port interface itself never import anything from
  `services/copernicus-engine.ts`, `services/copernicus-truth.ts`, or
  any other legacy engine, directly or by type reference — only the one
  legacy-wrapping adapter file does.

### 10.3 Sole implementation and deterministic fake provider

`legacy-copernicus-provider.ts` (inside `services/intelligence-adapters/io/`,
since it transitively touches `node:sqlite` via `copernicusForSite`),
wrapping `copernicusForSite(db, siteId, radiusKm, lookbackDaysFromWindow(request.temporalWindow), false)`
— the literal `false`, never omitted, never `satelliteValidationForSite`
(Section 3.2's trap) — and reshaping every field per Section 10.2's rules
before returning. `providerCode: "copernicus-legacy-simulated"` — the

import `copernicus-engine.ts`/`copernicus-truth.ts`.

`services/intelligence-adapters/satellite-fake-provider.ts` — a pure,
in-memory `SatelliteProviderPort` implementation for unit tests, distinct

observation" are handled downstream, in `satellite-observation-adapter.ts`
(a successful `fetch()` with zero usable scenes after adaptation, not a
provider-port-level failure) — see Section 17 (F-5) for the exact
aggregation rule this now feeds into.

### 10.3.1 Asynchronous port (post-review hardening, F-3)

**Decision: the provider port is asynchronous from Increment 10, not a
future migration.** `fetch(request: SatelliteProviderRequest): Promise<SatelliteProviderOutcome>`.

**Rationale:**

- Real satellite providers are network-bound; a synchronous signature
  could never accommodate one without a later breaking change.
- Timeout and rate-limit semantics (`unavailable.reason`) are inherently
  asynchronous concepts — modeling them behind a synchronous interface
  today would have been internally incoherent.
- Choosing asynchronous now avoids a future breaking port migration the
  moment a real provider is ever approved — the exact contradiction the
  independent review identified between this section's original claim
  ("supports a real provider later without redesign") and Section 4/14's
  original claim of synchronous, zero-redesign future aggregator
  compatibility. Those two claims could not both be true with a
  synchronous port; this decision resolves the tension by choosing
  asynchronous and updating the *other* claim instead (Section 4, Section
  14) rather than asserting both.
- The Satellite capability remains fully standalone in Increment 10
  (Section 6/19) — nothing about this decision touches Increment 9,
  which remains frozen, synchronous, and unchanged.
- **Future integration with Site Intelligence will require an explicitly
  authorized asynchronous composition increment, or an explicit
  precomputed-result boundary** (e.g. resolving the Satellite capability's
  own Promise into a plain value *before* handing it to the still-
  synchronous Increment 9 aggregator) — this is now stated as the actual
  future path, replacing the earlier, incorrect "already-compatible,
  zero-redesign" claim (Section 4).

**Consequences, stated explicitly so no implementation-time ambiguity
remains:**

- `satellite-intelligence-orchestrator.ts`'s own
  `getCanonicalSatelliteIntelligenceForSite(siteId)` is `async`, returning
  `Promise<SatelliteCapabilityOutcome>` — a genuine, disclosed divergence
  from Data Trust/Evidence Center's synchronous entry points.
- `satellite-intelligence-orchestrator-instance.ts` awaits the port call
  and returns the resolved `Promise`.
- **No `Promise.all` is used in Increment 10** — there is exactly one
  provider call per request; introducing concurrency machinery for a
  single awaited call would be unjustified complexity (Principle 17).
  Should a future increment add a second, independent provider call, that
  is the point to justify `Promise.all`, not before.
- Provider calls, if ever more than one is needed in a future increment,
  are awaited **sequentially** by default, matching the same
  no-premature-concurrency discipline Increment 9's own hardening
  established for its two synchronous capability calls.
- Timeout handling belongs at the provider adapter boundary (the one file
  that actually makes a real network call in any future real-provider
  implementation) — never inside the pure orchestrator core, which has no
  concept of wall-clock time beyond its own injected `now()`.
- `app/api/intelligence/satellite/site/handler.ts` is `async` (already
  required by Next.js route-handler convention regardless, matching every
  prior increment) and `await`s the orchestrator instance call.
- Route loading remains lazy — `route.ts` still dynamically `import()`s
  the instance module only after `requireAdminAuth` succeeds, unchanged
  by the sync-to-async decision.
- No frozen Increment 9 file changes as a result of this decision.

### 10.4 Evidence identity and missing-scene-id handling (post-review
hardening, F-4)

**Exact `EvidenceId` formula:**

```
satellite:<providerCode>:<siteId>:<sha256Hex(sourceSceneId.trim())>
```

Using the full, lowercase, 64-character SHA-256 hex digest of the
trimmed `sourceSceneId` (via `node:crypto`, the same accepted "pure
platform capability" precedent Increment 8's own checksum function
established) — never the raw scene id concatenated directly, so that an
arbitrary future provider's own scene-id format (which might contain
colons, slashes, or other characters unsafe inside an `Identifier`
string) can never produce a malformed or ambiguous `EvidenceId`, and so
that this formula can never collide with Evidence Center's own
`evidence:<siteId>:COPERNICUS` identifier (a completely different shape:
three colon-separated segments there, four here, with a 64-character hex
segment that a five-character type literal like `COPERNICUS` can never
match).

**`providerCode` is normalized** (trimmed, lowercased, matching the
literal value the provider port itself already declares, e.g.
`copernicus-legacy-simulated`) before being embedded — never taken from
unsanitized provider input.

**Missing `sourceSceneId` behavior — decided, not left open:** a
provider scene with `sourceSceneId: null` **cannot be adapted** into a
canonical `SatelliteObservation`/`Evidence` pair at all. It is excluded
from `observations`, and a `missing_scene_id` issue (`stage: "observation"`,
`severity: "significant"`, `canContinue: true`) is recorded, mirroring
the Evidence Adapter's own established "never fabricate an identifier;
report the gap instead" discipline. This scene contributes to the
"partial" aggregation case (Section 17, F-5) exactly like any other
rejected scene, never a silent omission.

The formula and every field it embeds (`providerCode`, `siteId`, the
scene-id digest) contains no credentials, no signed URL, and no raw
provider payload — confirmed by construction, since none of its three
inputs can ever contain any of those.

### 10.5 Neutral temporal-window request model (post-review hardening,
F-6)

`SatelliteProviderRequest.temporalWindow` replaces the earlier,
provider-shaped `lookbackDays` field:

```ts
export interface SatelliteTemporalWindow {
  readonly startAt: string;  // ISO-8601, UTC
  readonly endAt: string;    // ISO-8601, UTC; startAt <= endAt, enforced at request-validation time (Section 13)
  readonly maximumImageryAgeDays: number | null; // the canonical freshness threshold (Section 12), not a provider query parameter
}
```

**Rationale**: the original `lookbackDays` field mirrored
`copernicus-engine.ts`'s own `fetchSentinel1Metadata`/
`buildSentinel1SearchQuery` parameter naming and semantics almost
exactly — a genuine, if modest, leak of one provider's own query shape
into the supposedly neutral port. An explicit `[startAt, endAt]` window
is a strictly more general temporal-query model: any provider that
naturally thinks in "lookback N days" can trivially compute
`startAt = now - N days, endAt = now`, but a provider that naturally
thinks in explicit date ranges is not forced to awkwardly reverse-engineer
a "lookback" framing it doesn't use. `maximumImageryAgeDays` is kept as
its own, separate field — it is a **canonical quality/freshness rule**
(Section 12), never a provider query assumption; the port passes it
through only so a provider capable of server-side filtering may use it
as an optimization, never as the sole mechanism enforcing freshness
(freshness classification always happens canonically, downstream, per
Section 12, regardless of whether a given provider honors this hint).

**Timezone semantics are explicitly UTC** throughout — `startAt`/`endAt`/
`retrievedAt`/`captureTime`/`publishedAt` are all UTC ISO-8601 strings,
matching every existing timestamp convention in this codebase (no prior
increment uses a non-UTC timestamp anywhere).

**Default date-range behavior**, defined at the request-validation layer
(`handler.ts`, Section 13): when the caller does not supply an explicit
window, `endAt = now()` (the injected clock) and
`startAt = now() - defaultLookbackDays` — `defaultLookbackDays` sourced
from `config/copernicus_rules.json`'s own existing `defaultLookbackDays`
value, preserving the existing legacy default without introducing a new
one.

**Credentials discipline**: the legacy engine already never exposes
credentials (none exist to expose — `COPERNICUS_ACCESS_TOKEN`/
`COPERNICUS_CLIENT_ID` are read, if configured, only to be explicitly
*ignored* per the STAGE 0 TRUTH CORRECTION). A future real-provider
implementation must read credentials via `process.env` **only inside its
own outer, DB/network-touching module** (never the pure port interface,
never the orchestrator core, never evidence/observation payloads, never
logs) — stated here as a binding requirement for that future increment,
not something Increment 10 needs to enforce against code that doesn't
exist yet.

## 11. Imagery and binary handling decision

**Option A — metadata and evidence references only. Selected.**

Justification: matches the legacy engine's own existing, unmodified
`apiMode: "metadata_only"` / `allowDownload: false` configuration
exactly (`config/copernicus_rules.json`) — Increment 10 does not
introduce a new posture, it continues the existing one at the canonical
layer. `sourceUrl` (e.g. the OData catalog endpoint) is a **public**
catalog reference, not a signed or credentialed URL — safe to pass
through as a plain string reference, never a URL bearing a token or
temporary access grant. No imagery bytes are ever retrieved, held in
memory beyond a single request, or persisted. No binary asset table is
introduced. This keeps payload size bounded and predictable (bounded by
`rules.maxScenesPerSite = 5`), keeps test determinism trivial (no binary
fixtures), avoids licensing exposure (no image redistribution), and
avoids build/runtime cost or complexity a binary-handling capability
would add. Options B (byte processing), C (persistence), and D (signed
URLs) are all rejected — none is justified by any current requirement,
and each would meaningfully expand this increment's security and cost
surface for no identified benefit.

## 12. Deterministic intelligence rules

All outputs below are derived deterministically from source metadata
alone — no visual or CV judgment is made anywhere in Increment 10:

| Output | Deterministic derivation |
|---|---|
| Imagery available | `scenes.length > 0` after the provider call |
| Imagery age | `retrievedAt - captureTime`, in days, injected clock only |
| Freshness classification | `imageryAgeDays <= rules.scoring.recentSceneDays` (45, reused verbatim from `config/copernicus_rules.json`, not a newly invented threshold) → `"recent"`, else `"stale"` |
| Cloud/obstruction constraint | `cloudCoveragePercent: null` for the current SAR/Sentinel-1 source (post-review hardening, F-2: a provider-neutral nullable numeric field, not a Copernicus/SAR-specific enum — the legacy engine's own `cloudNote` field states this is a deterministic technical fact about SAR, not a per-scene visual assessment, and `null` honestly means "not reported by this source," equally usable for any future provider that also doesn't report it) |
| Spatial resolution | `spatialResolutionMeters: null` — not present in the legacy scene metadata today — reported as `null` with a disclosed limitation, never fabricated |
| Coverage match | `coordinateEligibility` from `evaluateCoordinateQuality.eligibleForSentinel`, reused verbatim |
| Usable-for-analysis classification | `qualityClassification !== "insufficient"`, itself mapped from the legacy engine's own `evidenceLevel`, not a new formula |
| Source quality classification | Direct mapping of the legacy `evidenceLevel` ("Alta"→"high", "Media"→"medium", "Baixa"→"low", "Insuficiente"→"insufficient") — a rename, not a new business rule, per `08_ADAPTER_STRATEGY.md`'s "adapters may translate shape and scale" rule |

**No visual or computer-vision observation is proposed by this plan.** Per
Section 5 and the mission's own preference, CV/ML inference is postponed:
no frozen CV capability exists anywhere in this repository to wrap
(confirmed by the directory audit, Section 3.1), and inventing new
inference logic inside an adapter would violate the binding "adapters
must not contain new business formulas" rule. This is deferred to a
future, separately-scoped increment (the mission's own suggested
Increment 13/14 range), to be designed only once a concrete, justified
model/service exists to wrap.

### HTTP mapping (post-review hardening, F-7 — finalized, no longer an
open decision)

The independent review explicitly challenged the original plan's uniform
"everything provider-related is 200" choice and asked that 503 be
evaluated as a candidate for transient provider failures. That challenge
is accepted: provider-access failures now map to 503 (Service
Unavailable — a semantically correct signal that an upstream dependency,
not this service's own request-handling, is the problem), and a
provider response that is reachable but structurally unusable now maps
to 502 (Bad Gateway — a semantically correct signal that the upstream
responded, but with something this service could not use), reserving 500
strictly for this capability's *own* code throwing, exactly matching
every prior increment's own "500 = our own bug" discipline.

| Condition | Aggregate status | HTTP |
|---|---|---|
| Invalid/missing `id`, or malformed `temporalWindow` | — | 400 |
| `requireAdminAuth` rejects | — | 401 or 503 (existing `AdminAuthResult` behavior, unchanged) |
| Both — i.e. the only — capability signal reports Site not found | `notFound` | 404 |
| Coordinates ineligible for satellite search | `unavailable` | **200** — the Site exists, the request was well-formed, and this is a truthful, disclosed non-result, not a provider or transport failure; kept as a deliberate, justified exception to the "unavailable → 5xx" pattern below, since nothing about a provider or network is even involved yet |
| Provider `misconfigured` / `invalid_credentials` | `unavailable` | **503** — not retryable without operator action (see retryability table below), but still correctly signals "this service could not obtain a result from its upstream dependency" |
| Provider `timeout` / `rate_limited` / `unexpected_error` | `unavailable` | **503** — retryable |
| Provider responded but every scene failed adaptation (all malformed / all missing scene id) | `unavailable` | **502** — the provider was reached and responded, but the response could not be used; distinct from 503 (upstream unreachable/erroring) and from 500 (this service's own bug) |
| `no_coverage`, or a successful response with zero scenes | `complete` (zero observations) | 200 |
| Some scenes usable, some rejected (mixed adaptation outcome, F-5) | `partial` | 200 |
| All scenes usable | `complete` | 200 |
| Orchestrator/handler/projection throws | `failed` | 500, sanitized, identical convention to Increments 7–9 |

**Why not 404 for missing imagery:** the Site *resource* exists; only
the satellite *observation* for it does not. Reusing 404 here would
conflate "the thing you asked about doesn't exist" with "the thing you
asked about exists but has no satellite data today" — two genuinely
different conditions this project's own established precedent (Increment
7/8's own site-not-found-vs-adaptation-issue distinction) already treats
as separate.

### Retryability (post-review hardening, F-7)

| Condition | Retryable |
|---|---|
| `timeout` | **true** |
| `rate_limited` | **true** |
| `unexpected_error` | **true** (unknown cause; allow retry) |
| `invalid_credentials` | **false** until configuration changes |
| `misconfigured` | **false** until configuration changes |
| Coordinates ineligible | **false** (deterministic; re-requesting cannot change eligibility) |
| `no_coverage` | **false** (not a failure; nothing to retry) |
| All scenes malformed | **false** (deterministic malformation of this specific request/data, not transient) |
| Stale imagery alone | **maybe** — retrying later could plausibly surface fresher imagery once the provider's own catalog updates; disclosed as informational only, never surfaced as a machine-actionable retry signal in the response body itself |

## 17. State matrix (post-review hardening, F-5)

### 17.1 Deterministic aggregate-status derivation rule

The independent review found the original plan's binary `success: boolean`
could not represent "some scenes in one batch adapted successfully, others
did not" — a real gap given Increment 8's own Evidence Orchestrator
precedent (`allEvidenceSucceeded`) explicitly handles the identical
per-item-mixed-outcome scenario. `SatelliteAggregateStatus` is now the
five-value enum introduced in Section 9 (`"complete" | "partial" |
"unavailable" | "failed" | "notFound"`), with this exact, deterministic,
mutually-exclusive, collectively-exhaustive derivation rule:

1. **`notFound`** — the Site itself does not exist. Short-circuits before
   any of the rules below are evaluated; no envelope is ever built for
   this case (Section 15).
2. **`failed`** — the capability's own internal processing threw an
   exception (a bug in the orchestrator, adapter, or projection itself)
   and no safe canonical result could be produced. This is the *only*
   status that indicates *our own code* broke, distinct from every case
   below, all of which indicate a normally-functioning capability
   reporting a real-world condition truthfully.
3. **`unavailable`** — zero usable observations exist, for any of three
   reasons: (a) coordinates are ineligible (`evaluateCoordinateQuality.eligibleForSentinel`
   is `false`, so the provider is never even queried), (b) the provider
   port itself returned `{kind: "unavailable", reason: ...}` (misconfigured,
   timeout, rate-limited, invalid credentials, or an unexpected error
   inside the provider adapter), or (c) the provider responded
   successfully with scenes, but **every single scene** failed adaptation
   (all malformed / all missing a `sourceSceneId` / all otherwise
   rejected) — no usable observation resulted despite the provider itself
   having been reachable.
4. **`partial`** — at least one usable observation exists, **and** at
   least one scene was rejected, stale, obstructed, incomplete, or
   otherwise limited. (Staleness alone, with no rejected scenes, does
   **not** trigger `partial` — see Group 7 below; freshness is
   informational, never an exclusion gate on its own.)
5. **`complete`** — every scene the provider returned (including the
   zero-scene `no_coverage` case, and the ordinary zero-scenes-attempted
   case) was either successfully adapted or there was nothing to adapt in
   the first place; no scene was rejected.

This ordering is evaluated top-to-bottom; the first matching rule wins,
guaranteeing no combination is ever ambiguous between two statuses.

**Site state** × **Provider state** × **Imagery state** × **Adaptation

| Group | Site | Provider | Imagery/Adaptation | Aggregate status | HTTP (finalized in Section 15, F-7) | Evidence included | Logging | Retry meaningful |
|---|---|---|---|---|---|---|---|---|
| 1 | not found | — | — | `notFound` | 404 | No | No | No |
| 2 | valid coords | — | — (coords ineligible before provider call) | `unavailable` | 200 | No | Yes — `satellite_coordinates_unavailable` | No (deterministic, re-requesting won't change eligibility) |
| 3 | invalid/missing coords | — | — | `unavailable` | 200 | No | Yes — `satellite_coordinates_unavailable` | No |

| 5 | valid, eligible | unavailable (`timeout`/`rate_limited`/`unexpected_error`) | — | `unavailable` | 503 | No | Yes — `satellite_provider_timeout` | Yes (transient) |

## 18. Data Trust interaction

- Satellite evidence uses the **existing canonical `Evidence` contract**
  (`services/intelligence/evidence/evidence.ts`), unmodified — not a new
  contract, only a new *adapter* producing it (Section 8).
- A **new evidence source classification is required**: `origin.source`
  (a `DataSourceId`) is set to a satellite-specific identifier (e.g.
  `"satellite-intelligence"`), distinct from Evidence Center's own
  `"evidence-center"` value — these are two independently-sourced
  evidence streams, both truthfully labeled, never conflated.
- **Satellite confidence is both source confidence and observation
  confidence, kept distinct**: `SatelliteQualityMetadata.sourceConfidence`
  (how much to trust the *provider's* data) is a separate field from the
  `Evidence.reliability` the evidence adapter computes (which, per
  Increment 5/8's own established discipline, is forced low
  — `COPERNICUS_RELIABILITY`-equivalent — to reflect the simulated-data
  invariant regardless of the legacy validation score's own value). The
  two are never merged into one number.
- **Satellite observations do not affect Data Trust in Increment 10.**
  Deferred, per Section 4's own table — no frozen Data Trust file is
  touched. Data Trust's own legacy formula already has an independent,
  unrelated, already-simulated satellite input via
  `confidence-engine.ts → satellite-validation-engine.ts`, which this
  increment does not touch, duplicate, or reconcile with.
- **The Satellite capability only publishes evidence for future
  consumers** — it does not attempt to wire itself into any consumer's
  formula. A future increment may choose to have a Recommendation
  Adapter or a Data Trust v2 cite satellite `EvidenceId`s; that is
  explicitly out of scope here (mirroring Increment 8's own identical
  deferral of wiring `EvidenceId`s into Score/Recommendation).

## 19. Evidence Center interaction

**No coupling of any kind.** Satellite Intelligence does not call
Evidence Center's orchestrator, route, or handler, and Evidence Center's
own frozen files are not modified. The two capabilities' evidence streams
both ultimately trace back to the same underlying legacy
`copernicus-engine.ts` data — a legacy-layer coincidence, not a canonical-
layer dependency, exactly mirroring how Evidence Center and Data Trust
already both independently reach `ensureDataTrustTables` without directly
depending on each other. **No circular dependency is possible**: Satellite
Intelligence never calls Site Intelligence (Increment 9), and Site
Intelligence does not call Satellite Intelligence in this increment
(Section 6). Satellite Intelligence does not mutate Data Trust output,
does not overwrite canonical site coordinates (it only *reads* them via
the Site Entity Adapter, exactly as Data Trust/Evidence Center already
do), and never silently upgrades confidence without a disclosed evidence
record backing it (Section 18).

## 21. Observability model

Stable, sanitized diagnostic codes (server-side only, `console.error`,
mirroring Increment 9's own post-audit-hardened F-3 convention exactly):

| Code | Fires when | HTTP it accompanies |
|---|---|---|
| `satellite_invalid_request` | (handled by 400 response; logging optional/omitted, matching precedent that 400s are not logged anywhere in this codebase today) | 400 |
| `satellite_coordinates_unavailable` | Site coordinates fail `evaluateCoordinateQuality.eligibleForSentinel` | 200 |
| `satellite_provider_misconfigured` | `reason: "misconfigured"` or `"invalid_credentials"` | 503 |
| `satellite_provider_timeout` | `reason: "timeout"`, `"rate_limited"`, or `"unexpected_error"` (grouped under one "transient" code — operators filtering for retryable causes want all three together; `misconfigured`/`invalid_credentials` are deliberately kept on their own, separate, non-retryable code above) | 503 |
| `satellite_no_coverage` | Provider outcome is `no_coverage`, or a successful response with zero scenes | 200 |
| `satellite_provider_adaptation_failed` | **One bounded, summary-level log entry** per request (post-review hardening, F-5) — fires whenever at least one scene was rejected, whether the outcome is `partial` (some usable) or `unavailable` (none usable); the single log entry carries a count of rejected scenes and their issue codes, **never one unbounded log line per malformed scene** | 200 (`partial`) or 502 (`unavailable`, all rejected) |
| `satellite_intelligence_failed` | Any exception in the orchestrator, handler, or projection step — kept as **one unified code**, deliberately not split into a separate "projection failed" code, matching Increment 9's own final, hardened convention exactly (Increment 9's post-audit hardening kept its analogous outer-catch code unified rather than distinguishing orchestrator-vs-projection failures, distinguishing instead only "own crash" from "aggregate-status-failed," which Satellite Intelligence already does via the separate codes above) | 500 |

**Logged fields, exhaustively**: diagnostic code, `providerCode`,
sanitized error `.name` only (never `.message`/`.stack`), capability
state, and — for `satellite_provider_adaptation_failed` only — a bounded
rejected-scene count and the set of distinct issue codes involved (never
per-scene raw content). **Never logged**: credentials, authorization
headers, signed URLs, raw provider response, imagery bytes (none are
ever retrieved), stack traces, error messages, user payload, secret
environment values, request coordinates beyond what a legitimate
diagnostic needs — identical discipline to Increment 9's own F-3
hardening, applied from the start this time rather than retrofitted.

**Duplicate logging policy**: layered, not deduplicated — mirroring
Increment 9's own explicit, audited conclusion that per-component plus
aggregate-summary logging is intentional observability, not noise
(confirmed acceptable by the Increment 9 final re-audit). A provider
failure logs once at the provider-outcome level; the capability's own
degraded `status` is disclosed in the response body, not re-logged
redundantly at the handler layer unless the handler's own code throws
(the `satellite_intelligence_failed` case above). Mixed-adaptation
failures (F-5) are explicitly bounded to one summary log line regardless
of how many individual scenes were rejected, preventing an unbounded log
volume from a single malicious or malformed provider response.

## 22. Side-effect guarantees

Increment 10 is read-only for business data, identically classified to
every prior increment: `copernicusForSite(..., false)` — the literal
`false`, never omitted, never `satelliteValidationForSite` (Section
3.2's trap) — skips every `INSERT` reachable from the legacy engine.
`ensureCopernicusTables(db)` remains a technically-reachable, one-time,
idempotent schema write on a fresh database, inherited legacy behavior,
not introduced by this increment (identical disclosure to Increments 7/8's
own `ensureDataTrustTables`/three-initializer-path classification).

**Explicitly prohibited, verified by the future test suite (Section 24)**:
`INSERT`/`UPDATE`/`DELETE`/`UPSERT`, any new `CREATE TABLE`/`ALTER TABLE`
beyond the already-existing, unconditionally-reached
`ensureCopernicusTables`, any filesystem write, any cache mutation
(**no caching is introduced — if ever wanted, it is explicitly a future,
separately-justified option, not part of this increment**), no
background job, no queue, no hidden telemetry write beyond the sanitized
`console.error` calls specified in Section 21.

Tests prove absence of side effects the same way Increments 7–9's own
side-effect sweeps do: a source-inspection regex sweep across every new
file for write-indicating patterns, plus explicit re-assertions that the
legacy Copernicus routes/tables remain reachable only through their own,
unchanged call sites.

## 29. Open decisions (post-review hardening, F-7/F-8 — resolved and

Every implementation-time decision this plan is responsible for making —
HTTP/retryability/diagnostic-code mapping (Section 15/17/21), the
provider port's neutrality and async signature (Section 10), the
`EvidenceId` formula (Section 10.4), the mixed-per-scene aggregation rule
(Section 17), the temporal request model (Section 10.5/13), the file and
port location split (Section 23), and the barrel-export timing (Section

remain open. What remains below is exclusively decisions that are
genuinely out of this increment's scope, deferred to a later, separately
authorized increment, exactly as ADR-020 and every prior increment's own
planning document already reserves this category of decision for:

1. **Selection of a real (non-simulated) satellite provider**, if and
   when one is ever approved — no API key, cost approval, licensing
   review, or vendor has been identified, and none is needed for
   Increment 10 (Section 10.3 wraps only the existing, already-approved
   legacy simulated engine).
2. **Provider licensing/commercial terms**, contingent entirely on (1) —
   not evaluable before a real provider is even chosen.
3. **Future caching strategy** — explicitly not introduced in Increment
   10 (Section 22); if ever wanted, it is its own future, separately
   justified design question.
4. **Future image/binary delivery** (bytes, signed URLs, tile services)
   — explicitly rejected for Increment 10 (Section 11); revisiting this
   is a future increment's own decision, contingent on a real,
   demonstrated need this plan has not identified.
5. **Future CV/ML interpretation** of satellite imagery — explicitly
   out of scope (Section 5/12); requires its own dedicated, versioned,
   evidence-traceable capability design once a concrete model/service
   exists to wrap, not a speculative placeholder here.
6. **Future multi-provider orchestration** (fallback between two or more
   live providers) — not meaningful to design before a second real
   provider exists; the port (Section 10) already supports adding one
   without redesign, but the *orchestration policy* across multiple
   simultaneous providers is a distinct future question.
7. **Future Site Intelligence (Increment 9) asynchronous integration
   authorization** — Section 10.3.1/Section 4 already establish that
   this requires an explicitly authorized asynchronous composition
   increment (or a precomputed-result boundary); *that future increment
   itself*, not the boundary condition triggering it, remains
   unscheduled and unauthorized.

**None of these seven remaining items blocks Increment 10
implementation.** Items 1–6 are contingent on an approval this increment
does not require (a provider, a budget, a caching/delivery/model
decision) and require no redesign of anything Increment 10 builds. Item
7 is the one exception, stated precisely rather than glossed over: it
*will* require its own future redesign work (an asynchronous composition
change to Increment 9's aggregator, or an explicit precomputed-result
boundary) — Increment 10's own architecture does not eliminate that cost,
it only establishes, honestly, exactly where and why that future cost
exists (Section 10.3.1), rather than asserting it away.

## 30. Final recommendation (post-review hardening — updated to reflect
F-1 through F-10 resolution)

The selected architecture — Option C: a fully independent capability,
wrapping the existing legacy simulated Copernicus/Sentinel-1 subsystem
behind a genuinely provider-neutral, asynchronous port, metadata-only,
read-only, with its own evidence stream and no coupling to Data Trust,
Evidence Center, or the frozen Site Intelligence Aggregator — **remains
approved**, unchanged by the hardening pass; every correction below
refined the architecture's own internal consistency, none reversed the
core decision.

- **F-1 through F-10 are resolved** (Section "Independent Review
  Hardening," below, for the full traceability record).
- **The truthfulness boundary is structurally enforced**: mandatory,
  non-optional `SatelliteTruthMetadata`, derived from the frozen
  `copernicusTruthMetadata()`, present on every non-`notFound` result and
  surviving projection unchanged (Section 9/14/15).
- **The provider abstraction is genuinely neutral and asynchronous**: no
  Copernicus/Sentinel-shaped type crosses the port; `fetch()` returns a
  `Promise`, deliberately chosen now rather than deferred into a future
  breaking change (Section 10).
- **The state matrix is deterministic**: a five-value aggregate status
  with an explicit, ordered derivation rule covering every combination,
  including the previously-missing mixed-per-scene case (Section 17).
- **HTTP behavior is finalized**: 400/401/503/404/200/502/503/500 mapped
  exhaustively and consistently across the contract, state matrix, and
  observability sections, with an explicit retryability table (Section
  15).
- **Evidence identity is finalized**: an exact, collision-safe formula,
  with missing-scene-id handling decided, not left to implementation-time
  judgment (Section 10.4).
- **The file inventory is finalized**: every path, including the
  provider port's own location and the two-file `io/` split, is a
  concrete decision, not a hedge (Section 23).
- **This plan itself does not authorize implementation.** Per this
  project's own established protocol (identical to every prior
  increment's planning → independent review → hardening → final
  focused re-review → freeze sequence), implementation authorization is
  granted only once a final focused re-review of this hardened document
  returns PASS — not by this document declaring itself ready.
